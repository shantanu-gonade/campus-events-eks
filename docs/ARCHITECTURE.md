## 1. System Overview

### Project Description
The Campus Events Management System is a cloud-native, microservices-based platform deployed on Amazon EKS (Elastic Kubernetes Service) that enables students, faculty, and staff to discover, create, and manage campus events. The system demonstrates modern DevOps practices, Infrastructure as Code (IaC), containerization, and comprehensive monitoring.

### Key Features
- **Event Management**: Create, update, delete, and search events
- **RSVP System**: Students can RSVP to events with capacity management
- **Real-time Updates**: WebSocket support for live event updates
- **Analytics Dashboard**: Event statistics and attendance analytics
- **Notifications**: Email/SMS notifications for event reminders and updates
- **Admin Dashboard**: Comprehensive analytics and event management
- **Responsive UI**: Mobile-friendly React application

### Design Principles
- **Microservices Architecture**: Loosely coupled services for independent scaling
- **Cloud-Native**: Built for AWS with EKS, RDS, and managed services
- **Infrastructure as Code**: Everything defined in Terraform
- **Security First**: Network policies, secrets management, encryption at rest and in transit
- **Observable**: Comprehensive monitoring with Prometheus and Grafana
- **Scalable**: Horizontal pod autoscaling and Karpenter node autoscaling
- **GitOps**: Git-based CI/CD with GitHub Actions

## 6. Infrastructure Architecture

### AWS Infrastructure Components

**VPC Configuration:**
- **CIDR Block**: 10.0.0.0/16
- **Availability Zones**: us-east-1a, us-east-1b, us-east-1c
- **Subnets**:
  - Public Subnets: 10.0.0.0/24, 10.0.1.0/24 (NAT Gateways, ALB)
  - Private Subnets: 10.0.2.0/24, 10.0.3.0/24 (EKS Nodes, Pods)
  - Database Subnets: 10.0.4.0/24, 10.0.5.0/24 (RDS)
- **Internet Gateway**: For public subnet internet access
- **NAT Gateways**: 2 (one per AZ for high availability)
- **Route Tables**: Separate for public and private subnets

**EKS Cluster:**
- **Version**: 1.31
- **Node Groups**: 
  - Initial: 2 t3.medium nodes (managed by Karpenter)
  - Auto-scaling: 2-10 nodes based on workload
- **Add-ons**:
  - EBS CSI Driver v1.53.0
  - VPC CNI (networking)
  - CoreDNS (service discovery)
  - kube-proxy (networking)

**RDS PostgreSQL:**
- **Engine**: PostgreSQL 16.3
- **Instance Class**: db.t3.micro (dev), db.t3.medium+ (prod)
- **Storage**: 20 GB GP3, autoscaling enabled
- **Multi-AZ**: Enabled for production
- **Backup**: 7-day retention
- **Encryption**: At rest with KMS
- **Parameter Groups**: Custom optimizations
- **Security**: Private subnets only, SG restrictions

**ECR Repositories:**
- `campus-events/frontend`
- `campus-events/events-api`
- `campus-events/notification-service`
- **Lifecycle Policies**: Keep last 10 images
- **Scan on Push**: Enabled

**Load Balancing:**
- **ALB**: campus-events-alb
- **Target Groups**: Auto-created by AWS LB Controller
- **Health Checks**: /health endpoint
- **SSL/TLS**: ACM certificate (production)
- **WAF**: Optional (production)

## 7. Security Architecture

### Network Security

**Security Groups:**
```
ALB Security Group:
- Inbound: 0.0.0.0/0:80, 0.0.0.0/0:443
- Outbound: EKS Worker Nodes

EKS Worker Nodes Security Group:
- Inbound: ALB on application ports
- Inbound: EKS Control Plane
- Outbound: Internet (via NAT)
- Outbound: RDS on 5432

RDS Security Group:
- Inbound: EKS Worker Nodes on 5432
- Outbound: None (database shouldn't initiate)
```

**Network Policies:**
```yaml
# Deny all traffic by default
kind: NetworkPolicy
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

# Frontend â†’ Events API
```

### Authentication & Authorization

**IAM Roles:**
- **EKS Cluster Role**: Manage EKS resources
- **EKS Node Role**: EC2 instances to call AWS APIs
- **ALB Controller Role**: Manage ALB resources
- **Karpenter Role**: Manage EC2 instances
- **Application Pods**: IRSA (IAM Roles for Service Accounts)

**RBAC (Kubernetes):**
```yaml
# Separate namespaces for isolation
- campus-events (application pods)
- monitoring (Prometheus, Grafana)
- karpenter (autoscaling)
- kube-system (system pods)

# Role-based access control
- Admins: Full cluster access
- Developers: Namespace-scoped access
- Services: Minimal required permissions
```

### Secrets Management

**AWS Secrets Manager:**
- Database credentials (master password)
- API keys (AWS SES, SNS)
- JWT signing keys
- External service credentials

**Kubernetes Secrets:**
- Mounted as environment variables
- Encrypted at rest with KMS
- Never committed to Git

**External Secrets Operator:**
- Syncs secrets from AWS Secrets Manager to Kubernetes
- Automatic rotation support
- Audit logging

### Data Security

**Encryption at Rest:**
- RDS: KMS encryption enabled
- EBS Volumes: Encrypted with KMS
- ECR Images: Encrypted in S3

**Encryption in Transit:**
- ALB â†’ Services: TLS 1.2+
- Service-to-Service: mTLS (optional with Istio)
- App â†’ RDS: SSL/TLS connections

**Data Privacy:**
- PII handling: Minimal data collection
- GDPR compliance considerations
- Data retention policies

## 8. Scalability & High Availability

### Horizontal Pod Autoscaling (HPA)

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: events-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: events-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

**Scaling Behavior:**
- Scales up quickly on traffic spikes
- Scales down gradually to avoid thrashing
- Based on CPU and memory metrics

### Node Autoscaling (Karpenter)

**Karpenter Configuration:**
```yaml
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: default
spec:
  limits:
    cpu: "100"
    memory: 200Gi
  template:
    spec:
      requirements:
      - key: karpenter.sh/capacity-type
        operator: In
        values: ["on-demand"]
      - key: kubernetes.io/arch
        operator: In
        values: ["amd64"]
      - key: node.kubernetes.io/instance-type
        operator: In
        values: ["t3.medium", "t3.large"]
```

**Scaling Triggers:**
- Pod pending due to insufficient resources
- Consolidation when nodes underutilized
- Spot instance interruption (future)

### High Availability Design

**Application Layer:**
- Multiple replicas of each service (min 2)
- Distributed across availability zones
- Load balanced with ALB
- Health checks and automatic restart

**Data Layer:**
- RDS Multi-AZ deployment
- Automated failover (< 2 minutes)
- Read replicas for read scaling (optional)
- Backup and point-in-time recovery

**Infrastructure Layer:**
- Multi-AZ VPC design
- NAT Gateway per AZ
- EKS control plane managed by AWS (HA built-in)
- ALB spans multiple AZs

### Deployment Strategy

**Rolling Update:**
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

**Characteristics:**
- Zero downtime deployments
- Gradual rollout (25% at a time)
- Automatic rollback on health check failures
- Preserves 2 old ReplicaSets for quick rollback

### Environment Strategy

**Development (dev):**
- Auto-deploy on merge to `develop` branch
- Minimal replicas (1-2)
- Smaller instance types
- Relaxed resource limits
- Debug logging enabled

**Staging (staging):**
- Manual promotion from dev
- Production-like configuration
- Same resource limits as prod
- Integration testing
- Performance testing

**Production (prod):**
- Manual promotion from staging
- High availability (min 2 replicas)
- Production instance types
- Strict resource limits
- Info-level logging
- Blue-green or canary deployments

## Conclusion

The Campus Events Management System demonstrates a production-ready, cloud-native architecture deployed on AWS EKS. The system incorporates modern DevOps practices including:

- **Microservices**: Loosely coupled, independently deployable services
- **Container Orchestration**: Kubernetes for container management
- **Infrastructure as Code**: Terraform for reproducible infrastructure
- **Continuous Delivery**: Automated CI/CD pipelines
- **Observability**: Comprehensive monitoring and logging
- **Security**: Defense in depth with multiple security layers
- **Scalability**: Automatic scaling at pod and node levels
- **High Availability**: Multi-AZ deployment for resilience

This architecture can handle production workloads while maintaining flexibility for future enhancements and scaling.
